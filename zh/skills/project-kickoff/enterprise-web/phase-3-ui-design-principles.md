# Phase 3 附录：UI/UX 设计原则

> **用途**: 企业级 Web 项目的前端交互设计指导  
> **版本**: v1.0  
> **来源**: 基于语音模型平台项目 UI Review 经验提炼  
> **适用对象**: AI Agent 在设计 UI 原型或前端页面时参考

---

## 1. 核心理念：以用户认知负担最小化为目标

### 1.1 角色分层体验

不同角色的用户拥有不同的技术背景和使用频率，界面设计应匹配角色特征：

| 角色类型 | 典型特征 | 设计策略 |
|----------|----------|----------|
| **终端用户 (End User)** | 非技术背景、使用频率低、目标明确 | 向导式引导、最小化选项、傻瓜式操作 |
| **管理员 (Admin)** | 有一定技术背景、使用频率中等 | 传统表格/表单、批量操作、筛选过滤 |
| **超级管理员 (Super Admin)** | 技术背景强、使用频率高 | 信息密度高、高级配置、系统级操作 |

**原则**：
- 终端用户不应该看到"管理后台式"的 Dashboard 作为首页
- 向终端用户展示的应该是**任务导向**的界面，而非**信息总览**式界面
- 管理员和超级管理员才需要 Dashboard 式首页（统计、监控、快捷操作）

### 1.2 信息分层

```
┌─────────────────────────────────────────────────────────────────┐
│                        信息分层金字塔                              │
│                                                                   │
│                         ┌───────┐                                │
│                         │ 引导页 │  ← 用户登录后首先看到           │
│                         │ 3个按钮│    只做决策，不做信息轰炸        │
│                         └───┬───┘                                │
│                      ┌──────┼──────┐                             │
│                      │      │      │                             │
│                 ┌────┴─┐ ┌─┴───┐ ┌┴─────┐                       │
│                 │创建  │ │查看 │ │控制台│  ← 按需进入功能         │
│                 │任务  │ │任务 │ │     │    分流，减少迷失感      │
│                 └──┬───┘ └─────┘ └─────┘                        │
│                    │                                              │
│            ┌───────┼───────┐                                     │
│            │ Step  │ Step  │  ← 向导式分步操作                    │
│            │  1    │  2... │    每次只展示一个决策                 │
│            └───────┴───────┘                                     │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 向导式交互 (Wizard Pattern)

### 2.1 何时使用

| 场景 | 推荐交互模式 | 原因 |
|------|:------------:|------|
| 终端用户创建/提交任务 | ✅ Wizard | 选项有先后依赖，逐步引导减少出错 |
| 终端用户填写申请/注册 | ✅ Wizard | 字段多时分步降低认知负担 |
| 管理员批量操作 | ❌ 传统表单 | 效率优先，管理员不需要引导 |
| 系统配置 | ❌ 传统表单 | 管理员操作，一次性设置 |
| 数据查询/筛选 | ❌ 传统表格+筛选器 | 信息浏览场景，wizard 不适合 |

### 2.2 设计要点

#### 分步原则 (Panel/Step 拆分)

```
✅ 好的拆分：每个 Step 只做一个决策
  Step 1: 选择芯片      (1 个选择)
  Step 2: 选择功能类型   (1 个选择，依赖 Step 1)
  Step 3: 选择语言       (1 个选择)
  Step 4: 输入唤醒词     (1 个输入)
  Step 5: 选择模型大小   (1 个多选 + 可选项)
  Step 6: 确认并提交     (汇总 Review)

❌ 坏的拆分：把所有表单项堆在一页
  → 一个长表单，用户看到 10 个字段就离开了
```

#### 视觉状态

每个 Panel 应有清晰的 3 种状态：

| 状态 | 视觉表现 | 交互行为 |
|------|----------|----------|
| **已完成 (Done)** | 收起，显示已选值 ✓ | 可点击展开回退编辑 |
| **当前 (Active)** | 展开，高亮边框 | 可操作、可前进/后退 |
| **锁定 (Locked)** | 收起，半透明 | 不可操作，等待前置步骤完成 |

#### 进度可见性

- 顶部始终显示进度条（步骤编号 + 当前位置）
- 已完成步骤变为绿色 ✓
- 用户始终知道"我在哪里、还有几步"

#### 预设值与智能默认

- 为每个选项设置合理默认值，用户可直接"下一步"
- 如果只有一个可选项（如芯片只授权了一个），自动选中并跳过或简化
- 最后一步提供完整摘要，让用户 Review 后一键提交

### 2.3 Wizard 的反模式

| ❌ 反模式 | ✅ 正确做法 |
|-----------|-------------|
| 每步太多字段（>3 个决策） | 每步聚焦 1 个核心决策 |
| 没有后退功能 | 允许回退到任意已完成步骤 |
| 最后没有确认摘要 | 提交前必须有 Review 步骤 |
| 提交后留在当前页 | 提交后弹出成功确认，引导下一步 |
| 步骤之间没有数据联动 | 后续步骤应根据前置选择动态调整 |

## 3. 登录后引导页模式 (Welcome / Launchpad)

### 3.1 适用场景

当终端用户的核心任务≤ 5 个时，使用引导页替代 Dashboard 作为登录后首页。

### 3.2 设计要点

| 要素 | 说明 |
|------|------|
| **操作按钮数量** | 3-5 个，对应用户最常用的任务 |
| **视觉权重** | 最核心的操作（如"创建任务"）视觉最突出 |
| **信息密度** | 极低 — 只展示操作入口，不展示详细数据 |
| **快捷统计** | 可在底部补充关键数字（如任务数），但不喧宾夺主 |
| **保留 Dashboard 入口** | 引导页的按钮之一应链接到完整 Dashboard |

### 3.3 与 Dashboard 的关系

```
引导页 (Launchpad)          Dashboard (控制台)
─────────────────          ─────────────────
面向：终端用户首次进入       面向：已熟悉系统或需要全局视图
目的：快速开始任务          目的：监控状态、查看统计
内容：3 个大按钮            内容：统计卡片 + 快捷操作 + 数据表格
```

## 4. 配置驱动的 UI 适配

### 4.1 原则

前端界面应能根据后端配置 **动态适配**，而非硬编码选项。

### 4.2 常见场景

| 场景 | 配置来源 | UI 行为 |
|------|----------|---------|
| model_sizes 只有一种 | `GET /system/options` | 自动选中，跳过该步骤或简化为确认 |
| 某芯片只支持一种功能 | 权限 + 芯片数据 | 自动选中功能类型，跳过选择 |
| 下载永不过期 | `model_download_expire_days = -1` | 不显示"过期时间"列 |
| 功能模块关闭 | `features.modules.xxx = false` | 隐藏对应导航和页面 |

### 4.3 实现指导

```
前端启动时：
  1. 调用 GET /api/v1/system/options 获取所有配置
  2. 缓存到 Pinia store
  3. UI 组件根据 store 数据动态渲染

动态跳过逻辑：
  if (enabledModelSizes.length === 1) {
    // 自动选中唯一选项
    // 在 Wizard 中该 Step 显示为"已确认"，不需要用户操作
  }
```

## 5. 检查清单

### 5.1 终端用户界面检查

- [ ] 登录后首页是引导页/Launchpad，而非 Dashboard
- [ ] 核心操作使用 Wizard 模式引导
- [ ] 每个 Wizard 步骤只包含 1 个核心决策
- [ ] 有合理的默认值/预设值
- [ ] 配置化选项只有 1 项时自动选中或跳过
- [ ] 不展示对用户无意义的字段（如后台配置值、内部 ID）
- [ ] 提交后有明确的成功反馈和下一步引导

### 5.2 管理员界面检查

- [ ] 使用传统表格/表单式布局（信息密度优先）
- [ ] 提供筛选、排序、批量操作
- [ ] 危险操作有二次确认
- [ ] 高级/危险操作收在折叠区域中

### 5.3 响应式与配置化检查

- [ ] UI 根据 system/options 接口动态渲染选项
- [ ] 功能关闭时对应 UI 隐藏
- [ ] 不存在硬编码的选项列表（芯片、功能类型、模型大小等）
